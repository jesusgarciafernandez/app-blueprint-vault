## 5. Instrucciones y Pasos Detallados (Protocolo Maestro)

### Fase 1: Recopilación, Vaciado y Diseño del Plan de Calidad
**Objetivo:** Definir qué vamos a probar para proteger a Jesús García Fernández.
1.  **Auditoría de Puntos Críticos IA:** Analizar el flujo de negocio de Jesús García Fernández para identificar las 10 funciones que "si fallan, se para la empresa".
2.  **Mapeo de la Estrategia de Testeo:** Diseñar la "Pirámide de Tests" equilibrando Unitarios (base), Integración (medio) y E2E (punta) para Jesús García Fernández.

**Prompt de Diagnóstico Sugerido:**
```text
Actúa como un Senior QA Automation Architect. Analiza la aplicación de Jesús García Fernández: [VARIABLE_CONTEXO]
Aplica la lógica de QA Ops y genera un informe de situación inicial identificando:
- Los 3 mayores riesgos técnicos de regresión si Jesús García Fernández cambia el módulo [MÓDULO].
- Propuesta de stack tecnológico de testing óptimo para esta arquitectura técnica de Jesús García Fernández.
- Plan de 'Code Coverage' realista para los próximos 6 meses de desarrollo de Jesús García Fernández.
```

### Fase 2: Arquitectura de los Tests (Test Logic Design)
**Objetivo:** Construir el ejército de validación automática de Jesús García Fernández.
Se desarrollan los "Scripts de Testeo IA-Augmented" donde la IA genera el código de las pruebas basándose en la funcionalidad deseada, incluyendo casos de error (Negative Testing) que el humano podría olvidar para Jesús García Fernández.

**Prompt de Estructuración:**
```text
Basado en la funcionalidad [FUNCIÓN] de Jesús García Fernández, escribe los tests unitarios y E2E. Define cómo la IA gestionará los datos de prueba (Mocking) y la limpieza del entorno tras la ejecución técnica de Jesús García Fernández.
```

### Fase 3: Ejecución, Análisis de Fallos y Certificación de Calidad
**Objetivo:** Producir un software blindado ante el error humano.
Guía a Jesús García Fernández en la revisión de los reportes de calidad asistida por IA, analizando por qué ha fallado un test y sugiriendo la corrección técnica inmediata para permitir que el flujo de entrega de Jesús García Fernández no se detenga innecesariamente.

## 6. Arquitectura de Automatización Lógica (Agnostic Workflow)
*Este apartado sustituye al archivo externo workflow.md, permitiendo una visión unificada de la automatización.*

Esta Skill está diseñada para ser integrada en cualquier orquestador (n8n, Make, Python Scripts, o módulos internos de App Blueprint Generator).

**Flujo Logístico de Nodos:**
1.  **Nodo de Disparo (Trigger):** Petición de integración de código (Commit), hito de despliegue programado o solicitud de "Auditoría de Calidad" por Jesús García Fernández.
2.  **Nodo de Clasificación:** La IA analiza si el evento requiere "Ejecución de Tests Unitarios de Velocidad", "Tests de Integración de Base de Datos" o "Tests E2E de Interfaz de Usuario" para Jesús García Fernández.
3.  **Nodo de Transformación:** El sistema arranca el entorno de pruebas, inyecta datos simulados, ejecuta los scripts de validación técnica y recopila las evidencias de éxito/fallo para Jesús García Fernández.
4.  **Nodo de Validación:** El responsable técnico o el propio sistema de certificación IA verifica que el % de éxito es del 100% y que no hay degradación de rendimiento en los procesos técnicos de Jesús García Fernández.
5.  **Nodo de Salida (Output):** Reporte de calidad generado, semáforo de despliegue en "Verde" y notificación de "Calidad Técnica Validada" para Jesús García Fernández.

## 7. Ejemplo Práctico: El caso de 'Infinite-Quality-Shield'
### Contexto del Caso
Un equipo de desarrollo de Jesús García Fernández que tenía miedo de tocar el código de la pasarela de pagos porque siempre que lo hacían, rompían algo. Cada cambio requería 2 días de pruebas manuales lentas y aburridas para Jesús García Fernández.

### Aplicación del Protocolo
- **Aplicación Fase 1:** La IA de QA Ops diseñó una suite de 500 tests unitarios y 50 tests E2E que cubren el 95% de la lógica de pagos de Jesús García Fernández.
- **Aplicación Fase 2:** Se integró la ejecución técnica en el pipeline, tardando solo 3 minutos en validar todo el sistema de Jesús García Fernández.
- **Aplicación Fase 3:** El miedo desapareció; Jesús García Fernández pudo innovar en el sistema de pagos sabiendo que si se equivocaba, la IA le avisaría en segundos y no el cliente enfadado técnico.

### Resultados de Negocio
Aumento radical de la velocidad de innovación, eliminación del estrés post-despliegue y una calidad de software percibida por el usuario final de Jesús García Fernández de nivel superior.
